集合

Java 集合概览

Java 集合(容器)，主要是由两大接口派生而来：Collection接口存放单一元素； Map 接口存放键值对。

List, Set, Queue, Map 的区别

一般地👇

  • List: 元素有序、可重复
  • Set: 元素无序、不可重复
  • Queue: 元素有序、可重复。按特定排队规则确定顺序
  • Map: 使用键值对存储，key 无序且不可重复; value 无序但可重复

集合功能接口

  • Iterable: 迭代器接口,实现类支持获取迭代器, 迭代集合元素
  • Collection: 单列集合的最父级接口
  • Map: 双列集合的最父级接口
  • SortedMap: 实现类可对key进行排序
  • List: 最常用的单列集合接口, 底层为Object[], 集合元素有序且可重复
  • Set: 实现此接口的集合, 元素一定不可重复, 但或可支持元素有序
  • SortedSet: 元素不可重复, 但可指定比较器比较集合元素, 实现元素排序
  • Queue: 队列结构的单列集合, 实现类有 单向出/入队列 的功能
  • Deque: 双端队列结构的单列集合, 实现类有 双向出/入队列 的功能
  • RandomAccess: 标识性接口, 实现了此接口的集合类支持快速随机元素访问

集合分类

  • 纯数组结构的集合, 元素有序,且支持快速随机访问, 例如ArrayList,Vector
  • 链表结构的集合, 元素有序, 增删效率较高, 例如LinkedList, LinkedHashSet
  • 树型结构的集合, 元素根据指定的比较器进行排序, 例如TreeSet, TreeMap(key排序)
  • Hash类集合, 元素不可重复, 例如HashSet, LInkedHashSet, HashMap(key不可重复)

线程安全的集合

JDK 提供的线程安全的集合类大部分都在 java.util.concurrent 包中

  • ConcurrentHashMap : 线程安全的 HashMap
  • CopyOnWriteArrayList : 线程安全的 List，在读多写少的场合性能非常好，远远好于 Vector
  • CopyOnWriteArraySet: 线程安全的Set
  • ConcurrentLinkedQueue : 高效的并发队列，使用链表实现。可以看做一个线程安全的 LinkedList，这是一个非阻塞队列
  • BlockingQueue : 这是一个接口，JDK 内部通过链表、数组等方式实现了这个接口。表示阻塞队列，适合用于作为数据共享的通道
  • ConcurrentSkipListMap : 跳表的实现。这是一个 Map，使用跳表的数据结构进行快速查找

HashSet去重原理

  1. HashSet通过元素的hashCode()和equals()方法校验是否已有重复元素
  2. 当元素添加进集合时, 先判断集合中是否存在相应的hash值, 如果不存在则允许入集合
  3. 如果待添加元素的hash值已存在, 则通过元素的equals()方法与已有元素的比较, 全部为false则允许入集合
  4. 如果hash值与equals()均未通过, 则拒绝入集合
  • 结论: 集合元素去重(检查元素是否重复), 必须正确重写hashCode()与equals()方法
  • hashCode()方法是为了提高检查重复的效率(如果hash值尚未在集合中存在, 则元素可直接入集合)
  • equals()方法是为了进行二次判断, hashCode()方法生成的hash值有碰撞风险

多线程

线程的生命周期与状态切换

什么是多线程以及为什么要用多线程

  1. 进程是一个程序的执行过程, 线程是相对进程来说的更轻量的执行单位, 每个进程都至少包含1个线程
  2. 相比于进程, 线程之间切换的资源开销小, 而且可以提高对硬件资源的利用率

  1. 新建态(new), 线程被创建后的初始状态
  2. 运行态(runnable), Java中将就绪态与运行时态统称为运行态
  3. 阻塞态(blocking), 线程等待锁释放时的状态
  4. 等待态(waiting), 与阻塞态类似, 此时线程任务处于暂停, 但是可被唤醒
  5. 超时等待态(time-waiting), 与等待态相比, 在指定的超时时间后会自动唤醒
  6. 终止态(terminated), 线程任务结束

join()实际上是Thread类的方法

线程的创建方式

  1. 继承Thread类, 覆写run()方法
  2. 实现Runnable接口, 覆写run()方法
  3. 实现Callable接口, 覆写call()方法
  4. 通过线程池创建线程, 降低系统资源消耗, 提高线程的响应速度和可管理性(推荐)

Q: 为什么要使用线程池来创建线程?

A: 能减少在创建和销毁线程上的时间及系统资源开销. 避免系统创建大量线程造成内存耗尽或者’过度切换’的问题

线程池

创建线程池

  • 线程池实现类ThreadPoolExecutor是 Executor 框架最核心的类
/**
 * 用给定的初始参数创建一个新的ThreadPoolExecutor。
 */
public ThreadPoolExecutor(
        // 线程池的核心线程数量.当线程池被创建后,无论是否有任务,核心线程会始终保持存活
        int corePoolSize,
        // 线程池的最大线程数.当任务队列已满(当前运行的线程数 = 核心线程数)时,当前可运行的线程数即变为最大线程数
        int maximumPoolSize,
        // 当线程数大于核心线程数时,多余的空闲线程存活的最长时间
        long keepAliveTime,
        // 上一项的时间单位
        TimeUnit unit,
        // 任务队列,用来储存等待执行任务的队列.
        // 当有新任务加入,但当前运行的线程数达到核心线程数时,新任务就会添加进任务队列
        BlockingQueue<Runnable> workQueue,
        // 线程工厂,用来创建线程,一般默认即可
        ThreadFactory threadFactory,
        // 拒绝策略,当提交的任务过多而不能及时处理时,我们可以定制策略来处理任务
        // 当有新任务加入,但当前运行的线程数等于最大线程数,且任务队列也已满时,
        // 就根据指定的拒绝策略对新任务进行处理
        RejectedExecutionHandler handler
) {

    // 参数合法性校验                   
    if (corePoolSize < 0 ||
            maximumPoolSize <= 0 ||
            maximumPoolSize < corePoolSize ||
            keepAliveTime < 0)
        throw new IllegalArgumentException();
    if (workQueue == null || threadFactory == null || handler == null)
        throw new NullPointerException();

    // 属性赋值
    this.corePoolSize = corePoolSize;
    this.maximumPoolSize = maximumPoolSize;
    this.workQueue = workQueue;
    this.keepAliveTime = unit.toNanos(keepAliveTime);
    this.threadFactory = threadFactory;
    this.handler = handler;
}

如何创建合理大小的线程池👇

有一个简单并且适用面比较广的公式：

  • CPU 密集型任务(N+1)： 这种任务消耗的主要是 CPU 资源，可以将线程数设置为 N（CPU 核心数）+1，比 CPU 核心数多出来的一个线程是为了防止线程偶发的缺页中断，或者其它原因导致的任务暂停而带来的影响。一旦任务暂停，CPU 就会处于空闲状态，而在这种情况下多出来的一个线程就可以充分利用 CPU 的空闲时间。
  • I/O 密集型任务(2N)： 这种任务应用起来，系统会用大部分的时间来处理 I/O 交互，而线程在处理 I/O 的时间段内不会占用 CPU 来处理，这时就可以将 CPU 交出给其它线程使用。因此在 I/O 密集型任务的应用中，我们可以多配置一些线程，具体的计算方法是 2N。

如何判断是 CPU 密集任务还是 IO 密集任务？

CPU 密集型简单理解就是利用 CPU 计算能力的任务比如你在内存中对大量数据进行排序。但凡涉及到网络读取，文件读取这类都是 IO 密集型，这类任务的特点是 CPU 计算耗费时间相比于等待 IO 操作完成的时间来说很少，大部分时间都花在了等待 IO 操作完成上

线程池饱和策略

如果有新任务要加入线程池, 但此时运行的线程数达到最大线程数, 且任务队列也已满, 则根据指定的拒绝策略来处理新任务

ThreadPoolTaskExecutor 定义了以下拒绝策略

  • ThreadPoolExecutor.AbortPolicy ：拒绝新任务并抛出异常, 默认
    • 抛出 RejectedExecutionException异常
  • ThreadPoolExecutor.CallerRunsPolicy ：调用执行自己的线程运行任务
    • 直接在调用execute方法的线程中运行被拒绝的任务. 性能低, 延迟高, 但是不会丢弃任务
  • ThreadPoolExecutor.DiscardPolicy ：直接丢弃新任务
  • ThreadPoolExecutor.DiscardOldestPolicy ： 丢弃最早未处理的任务

常见的线程池实现类

  • FixedThreadPool: 固定线程数的线程池, 核心线程数=最大线程数, 任务队列是LinkedBlockingQueue无界队列, 任务较多时可能会导致OOM(内存溢出)
  • SingleThreadExecutor: 只有一个线程的线程池, 核心线程数=最大线程数=1, 任务队列也是LinkedBlockingQueue无界队列, 任务较多时可能会导致OOM(内存溢出)
  • CachedThreadPool: 缓存型线程池, 根据需要创建线程执行任务. 核心线程数=0, 最大线程数=Integer.MAX.VALUE, 如果主线程提交任务的速度高于CachedThreadPool中线程处理任务的速度时，CachedThreadPool 会不断创建新的线程. 极端情况下会导致耗尽 cpu 和内存资源倒是OOM(内存溢出)

线程安全

并发编程的三个重要特性👇

  1. 原子性 : 一次操作或者多次操作，要么所有的操作全部都得到执行并且不会受到任何因素的干扰而中断，要么都不执行。synchronized 可以保证代码片段的原子性
  2. 可见性 ：当一个线程对共享变量进行了修改，那么另外的线程都是立即可以看到修改后的最新值。volatile 关键字可以保证共享变量的可见性
  3. 有序性 ：代码在执行的过程中的先后顺序，Java 在编译器以及运行期间的优化，代码的执行顺序未必就是编写代码时候的顺序。volatile 关键字可以禁止指令进行重排序优化

如何保证线程安全

使用线程安全的类; 利用synchronized/volatile关键字

  • synchronized 关键字可以修饰实例方法, 静态方法, 和代码块

  • synchronized 关键字加到 static 静态方法和 synchronized(class) 代码块上都是是给 Class 类上锁

  • synchronized 关键字加到实例方法上是给对象实例上锁

  • volatile 关键字是线程同步的轻量级实现, 性能比synchronized要好, 但只能修饰变量

  • volatile 关键字能保证数据的可见性和有序性,但不能保证数据的原子性.synchronized都能保证

  • volatile 关键字主要用于解决变量在多个线程之间的可见性, 而synchronized解决的是多个线程之间访问资源的同步性

SpringBoot

SpringBoot自动装配

Q: 什么是SpringBoot自动装配

A: SpringBoot 定义了一套接口规范, 在启动时会扫描外部 jar 包中的META-INF/spring.factories文件, 将配置信息加载到 Spring 容器, 并执行类中定义的各种操作.
对于外部 jar 来说, 只要按照 SpringBoot 定义的标准, 就能将自己的功能装配进 SpringBoot

Q: SpringBoot如何实现自动装配

A: 通过启动类上的核心注解SpringBootApplication实现. 这个注解包含了3个注解👇

  1. @SpringBootConfiguration: 本类为配置类, 允许在上下文中注册额外的 bean 或导入其他配置类
  2. @EnableAutoConfiguration: 启用SpringBoot自动装配, 是实现自动装配的重要注解
  3. @ComponentScan: 扫描被@Component类型注解标记的类, 默认扫描启动类所在包及其子包下的所有类

Spring事务

Spring管理事务的方式

  • 编程式事务 ： 在代码中硬编码 : 通过 TransactionTemplate或者 TransactionManager 手动管理事务，实际使用少
  • 声明式事务 ： 一般基于@Transactional 的全注解方式使用(推荐): 实际是通过 AOP 实现

Spring事务的传播途径

定义事务传播途径是为了解决业务层方法之间互相调用的事务问题

当事务方法被另一个事务方法调用时，必须指定事务应该如何传播. 例如, 方法可能继续在现有事务中运行，也可能开启一个新事务，并在自己的事务中运行

正确的事务传播行为可能的值如下:

1.TransactionDefinition.PROPAGATION_REQUIRED(默认)

如果当前存在事务, 则加入该事务; 如果当前没有事务, 则创建新事务

2.TransactionDefinition.PROPAGATION_REQUIRES_NEW

创建新事务, 如果当前存在事务, 则把当前事务挂起. 也就是说不管外部方法是否开启事务，Propagation.REQUIRES_NEW修饰的内部方法会新开启自己的事务，且开启的事务相互独立，互不干扰

3.TransactionDefinition.PROPAGATION_NESTED

如果当前存在事务，则创建一个事务作为当前事务的嵌套事务来运行；如果当前没有事务，则该取值等价于TransactionDefinition.PROPAGATION_REQUIRED

4.TransactionDefinition.PROPAGATION_MANDATORY

如果当前存在事务, 则加入该事务; 如果当前没有事务，则抛出异常

若是错误的配置以下 3 种事务传播行为，事务将不会发生回滚：
TransactionDefinition.PROPAGATION_SUPPORTS: 如果当前存在事务，则加入该事务；如果当前没有事务，则以非事务的方式继续运行
TransactionDefinition.PROPAGATION_NOT_SUPPORTED: 以非事务方式运行，如果当前存在事务，则把当前事务挂起
TransactionDefinition.PROPAGATION_NEVER: 以非事务方式运行，如果当前存在事务，则抛出异常

Spring事务的隔离级别

和事务传播行为这块一样，为了方便使用，Spring 也相应地定义了一个枚举类：Isolation

public enum Isolation {

    // 使用后端数据库默认的隔离级别. 例如MySQL默认REPEATABLE_READ隔离级别
    // 这一项也是默认项
    DEFAULT(TransactionDefinition.ISOLATION_DEFAULT),

    // 读未提交,最低的隔离级别,允许读取尚未提交的数据变更,可能会导致脏读,幻读和不可重复读
    READ_UNCOMMITTED(TransactionDefinition.ISOLATION_READ_UNCOMMITTED),

    // 读已提交, 允许读取并发事务已经提交的数据
    // 可以阻止脏读,但幻读或不可重复读仍有可能发生
    READ_COMMITTED(TransactionDefinition.ISOLATION_READ_COMMITTED),

    // 可重复读,对同一字段的多次读取结果一致,除非数据被本身事务所修改.
    // 可以阻止脏读和不可重复读,但幻读仍有可能发生
    REPEATABLE_READ(TransactionDefinition.ISOLATION_REPEATABLE_READ),

    // 序列化,最高的隔离级别,所有事务依次逐个执行,可阻止脏读,幻读和不可重复读,但会影响系统性能
    SERIALIZABLE(TransactionDefinition.ISOLATION_SERIALIZABLE);

    private final int value;

    Isolation(int value) {
        this.value = value;
    }

    public int value() {
        return this.value;
    }

}

Spring事务的回滚

**@Transactional(rollbackFor = Exception.class) **:

Exception 分为运行时异常 RuntimeException 和非运行时异常, 如果不配置rollbackFor属性,则默认遇到RuntimeException时才会回滚

当 @Transactional 注解作用于类上时，该类的所有 public 方法将都具有该类型的事务属性，同时，也可以在方法级别使用该标注来覆盖类级别的定义。如果类或者方法加了这个注解，那么这个类里面的方法抛出异常，数据就会回滚
